Response validation relied almost\nentirely on this ID. Together, this made DNS cache poisoning practical for an\nattacker who can observe one query or predict the next ID. The design\nconflicted with RFC 5452 recommendations for mitigating forged DNS answers.\n\n`inet_res` is intended for use in trusted network environments and with\ntrusted recursive resolvers. Earlier documentation did not clearly state this\ndeployment assumption, which could lead users to deploy the resolver in\nenvironments where faked DNS responses are possible.\n\nTherefore, the documentation is been updated to clarify that `inet_res` should\nonly be used in trusted networks and with trusted recursive resolvers.\n\nThe implementation is also improved to use strong random DNS transaction IDs\nand source ports for every DNS transaction. This should give ample protection\nagainst brute forcing fake DNS replies, known as DNS cache poisoning, but it\nstill does not protect against, for example, an adversary in the path of the\nDNS transaction that can observe the random values before faking malicious\nreplies, an attack known as DNS spoofing.\n\nFor randomization to happen, the Crypto application has to be loaded, which\nmost probably already should be the case for an Erlang node in an exposed\nnetwork.\n\nIf performance should become an issue, for applications within safe network\nenvironments, the previous light weight behaviour can be configured by setting\nthe resolver option `random` to `false`."
